# 01 - Philosophy: 工匠工具定位

> 技能系统不是 SaaS，不是平台，而是开发者的数字工具箱。

## TL;DR

- **定位**：技能系统是「工匠工具」而非「厂商产品」
- **关系**：工具 ↔ 工匠（平等合作），而非厂商 ↔ 用户（服务消费）
- **目标**：帮助开发者构建个人工作流，提升手艺
- **核心原则**：用户完全掌控、数据完全透明、工具永久归属

---

## 为什么是「工匠工具」？

### 软件开发者的真实处境

传统软件开发中，开发者面临两个困境：

| 困境 | 表现 | 后果 |
|-----|------|------|
| **工具割裂** | 每个项目一套脚本、一套约定 | 重复造轮子，无法沉淀经验 |
| **平台依赖** | SaaS 工具数据锁死、功能受限 | 失去掌控感，被动适应 |

开发者真正需要的不是另一个「平台」，而是一套**可积累、可定制、可带走**的数字工具。

### 工匠隐喻的价值

```
传统工匠：            数字工匠：
├─ 自己的工具箱  →    ├─ ~/.pi/agent/skills/
├─ 打磨趁手工具  →    ├─ 定制 skill 参数
├─ 传承手艺技法  →    ├─ 分享 skill 到社区
└─ 按自己节奏工作 →   └─ 完全本地运行，无网络依赖
```

**关键洞察**：当工具归属工匠时，工匠会主动维护、改进、传承工具。

---

## 核心价值主张

### 1. 可积累（Accumulative）

```yaml
# 开发者的技能资产随时间增长
year_1:
  skills: [git-helpers, file-utils]
  
year_3:
  skills: [git-helpers, file-utils, deploy-pipeline, code-review, ast-patterns]
  # 早期技能仍在使用，持续产生价值
  
year_5:
  skills: [..., k8s-expert, team-onboarding, custom-linter]
  # 形成个人独特的工作流体系
```

**对比**：SaaS 工具的技能不随你迁移，换工作/换项目从零开始。

### 2. 可定制（Customizable）

```yaml
# 技能根据你的习惯调整
skill: "@acme/deploy"
customize:
  default_env: "staging"        # 默认部署到 staging
  skip_confirm: true            # 跳过确认提示
  post_deploy: "notify-slack"   # 自定义钩子
```

**原则**：工具适应工匠，而非工匠适应工具。

### 3. 可带走（Portable）

```bash
# 你的工具箱跟随你到任何地方
$ pi skill export --all > my-toolkit.yml
$ scp my-toolkit.yml new-machine:~/
$ ssh new-machine "pi skill import my-toolkit.yml"
# 新机器瞬间拥有完整工作流
```

**对比**：SaaS 工具绑定账号，企业版更难迁移。

## 心理契约：工具 vs 厂商

| 维度 | 厂商产品 | 工匠工具 |
|-----|---------|---------|
| **关系** | 服务提供者与消费者 | 平等合作伙伴 |
| **目标** | 优化产品指标 | 提升工匠效率 |
| **控制权** | 厂商主导（功能、数据、定价） | 用户完全掌控 |
| **回报** | 产品更好用（被动受益） | 个人手艺精进（主动成长） |
| **归属感** | 你是「用户」 | 这是「你的工具」 |

### 文案设计示例

```yaml
# 厂商视角 ❌
install_message: "欢迎使用我们的智能技能平台"
update_message: "我们发布了新功能"
telemetry_message: "帮助我们改进产品"

# 工匠视角 ✅
install_message: "这把扳手现在属于你的工具箱"
update_message: "新增技法：快速定位死锁的 ast-grep 模式"
telemetry_message: "为你监测工具健康状况，预警潜在问题"
```

## 设计原则的落地

### 所有功能必须回答三个问题

1. **是否帮助工匠提升手艺？**
   - ❌ 采集数据用于产品分析
   - ✅ 分析代码模式，推荐改进建议

2. **工匠是否完全掌控？**
   - ❌ 云端强制更新
   - ✅ 本地可选升级，可锁定版本

3. **是否尊重工匠的现有投资？**
   - ❌ 新版本不兼容旧配置
   - ✅ 提供迁移工具和兼容层

### 反模式警示

#### 真实案例：SaaS 数据锁定的代价

> **场景**：某团队使用商业 CI/CD 平台，三年间在平台上积累了 500+ 条流水线配置、数千次部署历史、完整的测试报告。
>
> **问题**：
> - 平台宣布涨价 300%，迁移成本过高被迫接受
> - 尝试导出数据发现：配置只能导出为平台专有格式，无法在其他工具复用
> - 历史数据完全锁定在云端，导出需额外付费
>
> **教训**：当初如果选择可导出的、格式开放的工具，就不会被平台绑架。

#### 设计对比

```yaml
# ❌ 平台锁定
skill:
  cloud_required: true          # 必须联网
  data_storage: "vendor_cloud"  # 数据存储在厂商云端
  export_format: "proprietary"  # 专有格式，无法迁移
  
# ✅ 工匠友好
skill:
  offline_capable: true         # 完全离线运行
  data_location: "~/.pi/local"  # 数据本地存储
  export_format: "open_standard" # 标准格式，随时导出
  portable: true                # 配置可随工具箱迁移
```

## 社区与生态

### 工匠互助精神

```
我开发了一个 skill → 分享到社区 → 他人改进 → 回馈给我
         ↑                                      ↓
         └────────── 共同提升手艺 ──────────────┘
```

**开源不是做慈善，是手艺人的相互成就。**

### 互助如何落地：Telemetry 的工匠视角

详见 [04-evolution](./04-evolution.md#隐私与信任)，技能系统的 telemetry 设计体现了工匠互助的核心价值：

- **本地优先分析**：问题先在本地诊断，仅必要时才请求社区知识
- **完全透明**：用户随时查看 telemetry 日志，知道哪些数据被用于优化
- **工匠视角的文案**："帮你优化工作流"而非"帮助我们改进产品"

这种设计让数据真正服务于工匠群体的共同进步，而非厂商的产品指标。

### 技能质量的民主化

传统软件：质量由厂商把控（version 1.0 → 2.0）
技能生态：质量由社区定义（⭐ star、使用数据、改进建议）

最好的技能不一定是官方出品，而是被最多工匠验证、打磨的那一个。

---

## 总结

**技能系统的终极目标是**：

> 让每个开发者都能构建属于自己的、持续进化的、完全掌控的数字工具箱。

这不是一个产品，而是一种**工匠文化**的数字化表达。
